API documentation:

The backend exposes a simple REST API for managing tasks.  
All endpoints are available under the base route:

'/api/tasks'

Supported operations:

- **GET /api/tasks** – Returns the list of all tasks.
- **POST /api/tasks** – Creates a new task. Requires a 'title' in the request body.
- **PUT /api/tasks/:id** – Updates an existing task (title, description, or priority).
- **PATCH /api/tasks/:id/toggle** – Toggles the 'completed' status of a task.
- **DELETE /api/tasks/:id** – Deletes a task by its ID.

All requests and responses use JSON format.
---------------------------------------------------------------------------------------

Assumptions and design decisions: 


- Tasks are stored in memory on the backend, so data resets when the server restarts.
- Each task has a unique numeric ID generated by the backend.
- The 'title' field is required when creating or updating a task.
- The 'priority' field supports three values: 'low', 'medium', and 'high', with 'medium' used as the default.
- The frontend communicates with the backend through REST API calls using 'fetch'.
- All requests and responses use JSON format.
 API communication logic is centralized in the 'services/taskApi.js' file to separate network logic from UI components.
- The filtering logic is handled on the frontend using component state ('filter') rather than additional backend endpoints.
- Editing tasks is handled through controlled form state to allow updating task fields before sending them to the backend.
